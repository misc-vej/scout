<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Next.js middleware | Untrusted cookies/JWT cross here; middleware must validate, not trust |
| Next.js -> Supabase | Anon key + user JWT; RLS is the access-control boundary |
| auth.users trigger -> public.profiles | Privileged trigger writes a per-user row |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Elevation | profiles table | high | mitigate | Enable RLS in the same migration that creates the table; SELECT/UPDATE policies use `auth.uid() = id` so a user can only read/write their own row |
| T-01-02 | Spoofing | middleware session check | high | mitigate | Use `supabase.auth.getUser()` (validates JWT server-side), never `getSession()` (reads spoofable cookies) |
| T-01-03 | Info Disclosure | service vs anon key | high | mitigate | Only `NEXT_PUBLIC_SUPABASE_ANON_KEY` is referenced in code; service_role key never appears in repo or client bundle; `.env.local` is gitignored |
| T-01-04 | Tampering | handle_new_user trigger | medium | mitigate | Trigger is `SECURITY DEFINER` and only inserts (id, email) from `new`; no user-controlled column writes |
| T-01-05 | Elevation | unauthenticated route access | high | mitigate | Middleware redirects any unauthenticated request to /home,/beastiary,/discover,/profile to /auth |
| T-01-SC | Tampering | npm installs (@supabase/ssr, @supabase/supabase-js, @playwright/test) | low | mitigate | All packages are official, audited in RESEARCH.md §2; no [ASSUMED]/[SUS] packages introduced |
</threat_model>
