<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client form → Server Action (registerUser) | User-supplied email + password crosses here; must be validated/hashed server-side |
| Credentials authorize → Neon DB | Plain-text password compared against bcrypt hash; hash lives in DB |
| Middleware (Edge) → Protected routes | JWT token verified by NextAuth; no db access in Edge context |
| .env.local → Runtime | DATABASE_URL and AUTH_SECRET must never appear in client bundles |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | registerUser server action | mitigate | Verify caller is "use server" — Next.js enforces this; action is never reachable without the POST CSRF mechanism |
| T-01-02 | Tampering | users.passwordHash column | mitigate | bcrypt.hash(password, 12) in server action; plain-text password never written or logged |
| T-01-03 | Information Disclosure | AUTH_SECRET / DATABASE_URL | mitigate | Env vars in .env.local only (gitignored); no NEXT_PUBLIC_ prefix — never exposed to client bundle |
| T-01-04 | Elevation of Privilege | Middleware bypass | mitigate | Matcher pattern in middleware.ts covers all non-static paths; authConfig.authorized redirects unauthenticated users |
| T-01-05 | Denial of Service | registerUser called repeatedly | accept | Low-value target for v1; bcrypt cost 12 adds ~250ms server-side which naturally rate-limits; proper rate limiting deferred to v2 |
| T-01-SC | Tampering | npm install (next-auth@beta, @auth/drizzle-adapter, bcryptjs, drizzle-orm, @neondatabase/serverless, drizzle-kit) | mitigate | Packages validated in RESEARCH.md Package Legitimacy Audit; all are [CONFIRMED] mainstream packages |
</threat_model>
